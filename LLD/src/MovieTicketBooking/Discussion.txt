/*
Requirements :

1. User can search for movies by title.
2. User can browse movies playing at a given theatre.
3. Theatres have multiple screens. All screens have same seating arrangement.
4. User can see available seats for a showtime and select specific ones.
5. User can book multiple available seats in a single transaction; booking returns a confirmation ID.
6. Concurrent booking of the same seat; only one succeeds.
7. User can cancel a reservation by confirmation ID, releasing the seat.

Out Of Scope :

1. Payment Processing
2. variable seats layout
3. Food meal AddOns
4. Rescheduling  (cancel and rebook instead)
5. UI / rendering


Entity : 

1. BookingSystem (class) - Orchestrator. Owns theater. Entry point for search and cross-theater queries.

2. Movie (class) - Searchable entity. Title plus ID. Ties showtimes together across theaters.

3. Theater (class) - Named location. Owns a list of showtimes.

4. Showtime (class) - A specific screening. Tracks per-seat availability and handles booking 
                      concurrency.

5. Reservation (class) - User's booking reference. Stores confirmation ID and which seats were booked
                         used for cancellation.

6. Seat (String) - A string identifying a specific seat, e.g. "A5". No behavior, just an identifier.

7. Screen (String) - Label identifying which room a showtime is in, e.g. "Screen 3".

Key - Relationships :

1. BookingSystem -> List<Theater>
2. Theater -> List<Showtime>
3. Showtime -> Theater (back-reference for navigation)
4. Showtime -> List<Reservation> (booking records for this showtime)
5. Reservation -> Showtime (back-reference for cancellation routing)
6. Reservation -> List<String> (e.g. ["A5", "B4"])

*/
